## build.py hash randomization (RESOLVED)

DISCOVERED 2026-07-21 during division-field migration testing — build.py uses Python built-in hash(slug) % 8 (lines ~75 and ~259) to pick which of 8 rotating H1/title/meta-description templates a page gets. Python randomizes string hash() per process by default (PYTHONHASHSEED), so this index changes on every single build run regardless of any code or data change. Proven experimentally: two consecutive builds of identical, unchanged code produced 41,367 of 47,313 differing output files. IMPACT: every rebuild silently shuffles meta descriptions/titles/H1s sitewide at random; the live site currently reflects whatever random variant the last deploy happened to produce, not a deliberate choice; deploys are not reproducible. NOT YET FIXED. Recommended fix: replace hash(slug) with a stable hash (e.g. int(hashlib.md5(slug.encode()).hexdigest(), 16) % 8) so template selection is deterministic per-slug across all runs.

Fix deployed 2026-07-21: replaced hash(slug) with stable_idx() using hashlib.md5, verified deterministic across 3 consecutive builds (0 diffs), deployed live, all deploy.sh verification stages passed (homepage 200, robots.txt 200, sitemap.xml 200, both contact forms 302 + no missing-fields error).
